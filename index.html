Erire Current Language Reference

This document describes the language and tooling supported by the current runtime in this repository. It is intentionally grounded in the current parser, semantic analyzer, runtime, UI layer, and packager.

1. Mental Model

An Erire program is a sequence of statements that runs in source order.

  • UI is created with screen.add[...].
  • Runtime state is stored with var.set[...].
  • Inputs can push text into variables through input.bind[...].
  • Event blocks run later in response to user actions.
  • backend.er is loaded automatically before the entry file when it sits beside it.

Important runtime behavior:

  • screen.add[...] creates a node immediately.
  • screen.setText[...] only works after a node with that id exists.
  • onLoad[...] is deferred until the UI has been created.
  • Accessing an undefined variable currently resolves to an empty string.

2. Program Structure

The two most common project shapes are:

Small CLI-created project

MyApp/
  main.er
  backend.er
  app.json
  assets/
  python/
  cpp/

Larger Studio-style project

MyApp/
  app.json
  src/
    main.er
    backend.er
  pages/
  components/
  assets/
  python/
  build/
  tests/

3. Supported Statements

screen.create[...]

Creates or configures the app window.

Accepted shapes today:

screen.create[100; 100; 900; 560]
screen.create[app; size; 100; 100; 900; 560]

The six-argument form is the one used by most current examples.

screen.title[...]

Sets the window title.

screen.title["My App"]

screen.bg[...]

Sets the window background color.

screen.bg["#0f172a"]

screen.add[...]

Adds one UI element to the active app.

screen.add[text.value["Hello"]]

screen.show[...]

Switches the current page.

screen.show["settings"]

The runtime starts on page main. If you place nodes on another page such as home, call screen.show["home"] during startup.

screen.setText[...]

Updates the text/value of an existing node by id.

screen.setText["headline"; "Updated"]

This is also how current Erire changes:

  • text node content
  • input content
  • button labels
  • webview URLs by changing the webview node text

var.set[...]

Creates or updates a runtime variable.

var.set["status"; "Ready"]
var.set["sum"; math.add[2; 3]]

import python "...py" as alias

Registers a Python bridge script.

import python "./python/app_bridge.py" as app

The active import system used in real apps today is the Python import form above.

Control Flow

Erire currently supports:

if[$count > 0][
    screen.setText["status"; "Positive"]
] else.if[$count == 0][
    screen.setText["status"; "Zero"]
] else[
    screen.setText["status"; "Negative"]
]

while[$counter <= 5][
    var.set["counter"; math.add[$counter; 1]]
]

for["i"; 1; 4][
    screen.setText["status"; $i]
]

for[...] accepts:

for["name"; start; end]
for["name"; start; end; step]

4. Values and Expressions

Erire values can be:

  • strings: "Hello"
  • numbers: 42
  • booleans: true, false
  • symbols: app
  • variables: $title, $result.message, $result.items.0
  • call expressions: math.mul[6; 7], py.call["app.fetch"]

Structured JSON returned from Python is flattened into dotted variable paths. If Python returns:

{ "message": "Done", "stats": { "count": 3 } }

Then Erire can read:

$result.message
$result.stats.count

5. Elements

text / label

Text output nodes.

Common usage:

screen.add[text.value["Hello"].size[24].color["#ffffff"]]

btn / button

Clickable button nodes.

screen.add[button.text["Save"].onClick[
    screen.setText["status"; "Saved"]
]]

input

Text input fields, including multiline and read-only variants.

screen.add[input.placeholder["Type here"].bind["query"]]

image

Bitmap/image nodes.

screen.add[image.src["./assets/logo.png"].fit["contain"]]

webview

Embedded browser node. A URL is required.

screen.add[webview.url["https://example.com"].id["browser"]]

box / card / panel

Surface/container-like nodes for custom-drawn panels and cards.

screen.add[card.text["Overview"].radius[20].bg["#111827"]]

card and panel are specialized aliases of box with different default surface styles.

6. Common Properties

These properties are shared by most elements:

| Property | Meaning | | --- | --- | | x, y | Position | | w, h | Size | | id | Node id for later screen.setText[...] or command targeting | | page | Page name used by screen.show[...] | | preset | Built-in style preset | | color | Text color | | bg | Main background color | | bg2 | Secondary background color for painted surfaces | | border | Border width | | borderColor | Border color | | radius | Corner radius | | padding | Inner spacing | | shadow | Shadow size | | shadowColor | Shadow color | | align | Text alignment: left/start, center/middle, right/end | | onLoad | Event block executed after app creation |

Notes:

  • Colors currently use #RRGGBB.
  • border is a width, not a color.
  • Absolute positioning is still the main layout model today.

8. Events and Bindings

onClick[...]

Runs when a button or clickable node is activated.

screen.add[button.text["Refresh"].onClick[
    screen.setText["status"; "Refreshing"]
]]

onChange[...]

Runs when an input changes.

screen.add[input.bind["query"].onChange[
    screen.setText["status"; $query]
]]

onLoad[...]

Runs after the UI tree has been created. This is the safest place to do initial py.call[...] setup that fills the interface.

screen.add[text.value[""].id["boot"].onLoad[
    var.set["snapshot"; py.call["app.bootstrap"]]
    screen.setText["status"; $snapshot.message]
]]

input.bind[...]

bind links an input to a runtime variable.

Behavior today:

  • the initial input text is copied into the bound variable
  • on every change, the variable is updated before the onChange[...] block runs

9. Built-In Native Calls

These are the current native value calls accepted by semantic analysis and executed by the runtime.

System and Time

  • time.now[]
  • sys.cwd[]
  • sys.hostname[]
  • sys.platform[]

Text

  • text.upper[value]
  • text.lower[value]
  • text.title[value]
  • text.length[value]
  • text.contains[text; needle]
  • text.concat[a; b; c; ...]

Math

  • math.add[...]
  • math.sub[...]
  • math.mul[...]
  • math.div[...]
  • math.min[...]
  • math.max[...]

10. Python Bridge

Python bridge calls use:

py.call["alias.function"; arg1; arg2; ...]

Example:

import python "./python/app_bridge.py" as app
var.set["result"; py.call["app.ping"]]
screen.setText["status"; $result.message]

Current bridge rules:

  • target format must be alias.function
  • the alias must have been registered with import python
  • arguments are JSON-serialized, then base64-encoded
  • Erire invokes the script with:
    • --erire-call <function>
    • --erire-args-b64 <encoded-json-list>
  • the script must print JSON or a scalar-like payload to stdout

When Python returns structured JSON, Erire stores both the root variable and the dotted child paths.

11. Generic Commands

These are the non-statement commands currently supported by semantic analysis and runtime execution.

WebView Commands

webview.open["browser"; "https://example.com"]
webview.back["browser"]
webview.forward["browser"]
webview.reload["browser"]
webview.runJS["browser"; "document.title='Updated'"]

Window Commands

screen.icon["./assets/logo.png"]
window.icon["./assets/logo.png"]
window.caption["Another Title"]

Console Commands

console.open["powershell"]

Built-in profiles include powershell, cmd, wsl, ubuntu, debian, kali, and devshell. Custom profiles can be added in tools/console_profiles.txt.

Animation Commands

anim.play["hero"; "float"; 1800; "loop"]
anim.oscillate["hero"; "y"; 18; 1800; "loop"]
anim.keyframes["hero"; "x"; "0:0,0.5:20,1:0"; 1400; "loop"]
anim.stop["hero"]

These commands are implemented for the Windows-first runtime path.

13. app.json

app.json is used by runtime, Studio, and packaging tooling.

Common fields seen in this repository:

{
  "name": "My App",
  "entry": "src/main.er",
  "output": "build/my_app.exe",
  "icon": "assets/brand/app.ico",
  "win_title": "My App",
  "win_icon": "assets/brand/app.ico",
  "python": "python/app_bridge.py",
  "type": "desktop-ui",
  "template": "workspace",
  "description": "Structured workspace tool",
  "version": "0.1.0"
}

Current behavior by field:

  • name: used by runtime as a fallback title and by tooling metadata
  • entry: used by Studio/project tooling
  • output: used by template/tooling metadata
  • icon: runtime + packager icon path
  • win_title: runtime and packaging title override
  • win_icon: runtime and packaging icon override
  • python, type, template, description, version: tooling metadata used mainly by Studio templates and project workflow

Important current nuance:

  • the runtime and packager look for app.json beside the entry .er file
  • Studio project discovery uses a root-level app.json

If your project entry lives in src/main.er and you want runtime/packager metadata such as win_title or win_icon honored today, keep that metadata in the entry directory or align your project layout accordingly.

14. Packaging Behavior

erire --build ... packages:

  • the entry .er file
  • auto-loaded backend.er
  • imported Python bridges
  • assets/
  • python/
  • app.json
  • optional compiled helper from cpp/

Important packaging notes:

  • Python bridge packaging requires PyInstaller.
  • PNG icons are converted through tools/png_to_ico.py.
  • The packaged app is appended to ErireRunner.exe.
  • At runtime, the packaged payload is extracted to a temp directory before execution.

15. Limits and Non-Goals of the Current Runtime

  • Layout is still explicit and coordinate-based.
  • The component/layout/state system described in erire-next-language.md is not the current shipped language.
  • The runtime is optimized around desktop UI, not general-purpose scripting.
  • The implementation is Windows-first, even though a few interfaces have fallback stubs.

For app architecture guidance rather than syntax details, continue with Building Full Apps with Erire.

Building Full Apps with Erire

This guide explains how experienced developers should think about building a complete desktop application with Erire as it exists today.

The short version:

  • use Erire for the desktop UI shell
  • use backend.er for lightweight app logic
  • use Python for data, files, automation, APIs, and non-trivial processing
  • use assets/, app.json, and packaging from the start
  • grow the project in the Studio-style structure even if you are not using Studio yet

1. Choose The Right Project Shape

For serious work, use a structured layout:

MyApp/
  app.json
  src/
    main.er
    backend.er
  pages/
  components/
  assets/
  python/
  build/
  tests/

Why this shape works well:

  • src/main.er owns the screen and event wiring
  • src/backend.er owns native Erire setup and derived values
  • pages/ holds page-specific ideas even before Erire gets full component imports
  • assets/ is ready for images, icons, and packaged resources
  • python/ gives the app a real backend layer
  • build/ keeps outputs out of source folders

Even if the runtime does not yet import pages/*.er as first-class Erire modules, this structure keeps the project sane as it grows.

One current implementation detail to plan around:

  • Studio uses a root app.json for project discovery
  • runtime and packaging metadata are currently read beside the entry .er file

So if your entry is src/main.er and you rely on runtime metadata like win_title or win_icon, be deliberate about where that metadata lives.

2. Split Logic Into Three Layers

Layer A: UI Composition in main.er

Keep main.er focused on:

  • window creation
  • page shells
  • ids
  • event wiring
  • calls to screen.show[...]
  • calls to screen.setText[...]

Layer B: Lightweight Native Logic in backend.er

Use backend.er for:

  • startup values
  • string formatting
  • math
  • native loops
  • simple state derivation

This keeps obvious setup logic inside Erire itself.

Layer C: Heavy Logic in python/

Use Python when the app needs:

  • filesystem access
  • JSON and config handling
  • HTTP or external services
  • search/indexing
  • automation
  • system inspection
  • data transformation

This keeps the UI layer small and readable while still letting the app do real work.

3. Design The App Before Writing Widgets

Because current Erire layout is coordinate-driven, planning matters.

Before coding:

  1. List your pages.
  2. Decide which ids will be updated later.
  3. Decide which inputs need bind[...].
  4. Decide which actions stay native and which go to Python.
  5. Sketch the rectangles and spacing system you will use.

Good questions to answer first:

  • What is the home page?
  • Which controls need dynamic updates?
  • Which results are shown as text, lists, or embedded web pages?
  • What should load in onLoad[...]?

4. Build One Stable Screen Shell First

In current Erire, strong apps usually start with:

  • title area
  • content area
  • status area
  • primary action buttons

That shell gives you stable ids and stable geometry before you layer in data.

Recommended shell ids:

  • page_title
  • page_subtitle
  • status
  • primary_input
  • result_view

Keep ids predictable and descriptive. screen.setText[...] is one of the main update paths in the runtime, so id discipline is part of app architecture.

5. Treat Pages As Distinct UI States

Use page[...] and screen.show[...] to separate workflows cleanly.

Typical page split:

  • overview
  • editor
  • logs
  • settings
  • help

This is better than trying to cram every control into one giant screen and manually hide pieces with text tricks.

6. Use bind[...] For Form Inputs

Inputs with bind[...] are one of the most practical tools in current Erire.

Use them for:

  • search boxes
  • form fields
  • filenames
  • filters
  • URLs

Then let event blocks consume the variable:

screen.add[input.bind["query"].id["queryInput"].onChange[
    screen.setText["status"; $query]
]]

This keeps the UI readable and reduces manual input plumbing.

7. Push Data Work To Python Early

A common Erire mistake is trying to force too much non-UI behavior into screen.setText[...] chains.

Better approach:

  • Erire owns the interaction
  • Python owns the expensive or stateful work
  • Erire receives a clean result payload

Example flow:

  1. User clicks Load Project.
  2. Erire calls py.call["app.scan_project"; $project_root].
  3. Python returns a dictionary with counts, labels, and paths.
  4. Erire updates several nodes from $scan.count, $scan.summary, $scan.root.

That pattern scales much better than building pseudo-business logic inside the UI file.

8. Use Structured Python Results Deliberately

Erire stores structured JSON into dotted variables automatically.

If Python returns:

{
  "message": "Indexed",
  "stats": { "files": 42, "errors": 0 }
}

Then your UI can read:

$result.message
$result.stats.files
$result.stats.errors

Design your Python responses for UI consumption:

  • message for a quick status line
  • title for a heading
  • items for content
  • stats for metrics
  • error for failures

9. Keep A Packaging Mindset From Day One

If the final goal is a packaged .exe, build with packaging in mind:

  • keep assets under assets/
  • keep helper scripts under python/
  • keep icons in assets/brand/
  • keep metadata in app.json

Current packager behavior already rewards this structure.

If your app imports Python:

  • install PyInstaller
  • make sure the bridge script follows the Erire CLI bridge contract
  • keep helper dependencies explicit

If your app needs native compiled code:

  • place C++ sources under cpp/
  • let the packager compile and bundle the helper

10. Prefer Studio Templates For Team-Scale Work

Erire Studio templates already encode good structure:

  • default
  • blank
  • workspace
  • python-tool
  • showcase

Good rule of thumb:

  • use blank for tiny prototypes
  • use workspace for internal tools and dashboards
  • use python-tool when bridge logic is central
  • use showcase for presentation/demo apps

These templates are not just convenience; they are working guidance for how the project expects apps to grow.

11. A Good Real-World Blueprint

Here is a solid architecture for a complete Erire app:

App Type

A local document or project workspace tool.

UI Layer

  • left navigation
  • center workspace
  • bottom status panel
  • secondary settings page

Native Erire Layer

  • startup text
  • basic counts and labels
  • button click wiring
  • page navigation

Python Layer

  • project scanning
  • reading/writing JSON
  • asset indexing
  • search helpers
  • export/build helpers

Packaging Layer

  • app.json
  • assets/brand/app.ico
  • python/app_bridge.py
  • final build in build/

This is exactly the kind of app Erire is already good at.

12. Design Rules That Work Well Today

  • Keep main.er readable enough to scan top-to-bottom.
  • Prefer a small number of stable ids over random one-off ids.
  • Use card, panel, and presets to build consistent visual sections.
  • Put startup fetches inside onLoad[...], not above the screen tree.
  • Avoid gigantic event blocks; hand off work to Python quickly.
  • Treat absolute layout like a design system, not a pile of magic numbers.

13. Release Checklist

Before packaging:

  1. Run erire --check.
  2. Verify every dynamic screen.setText[...] target exists.
  3. Verify every Python import path is correct.
  4. Verify app.json title/icon metadata.
  5. Verify assets live under assets/.
  6. Install PyInstaller if Python is used.
  7. Run erire --build.
  8. Launch the packaged .exe and verify the real packaged flow.

14. Final Advice

The best complete Erire apps are not the ones that try to make Erire do everything. They are the ones that let Erire stay what it is best at:

  • desktop UI
  • explicit flows
  • direct event wiring
  • clean handoff to Python when the work stops being UI work

That separation gives you a system that stays small, understandable, and shippable.

Erire Dev Log

This dev log is a repository-based milestone log. It is not reconstructed from git history; it is based on the current codebase, shipped examples, templates, and supporting docs.

Snapshot

  • Current CLI banner in src/main.c identifies the tool as Erire 1.0.4.
  • The project currently includes the runtime, packager, examples, tests, and Erire Studio.

Milestone 1: Core Language and Frontend Pipeline

The first visible milestone is the core frontend pipeline:

  • lexer
  • parser
  • AST
  • semantic analyzer
  • runtime execution

This established the basic Erire shape:

  • chain-based element syntax
  • screen.* commands
  • var.set[...]
  • values, calls, variables, and conditions

At this point, Erire already had a strong identity: it was not trying to be a general programming language. It was trying to be a UI-first desktop language.

Milestone 2: Native Desktop UI Runtime

The runtime and UI layer evolved into a native desktop shell with these node families:

  • text/label
  • button
  • input
  • image
  • webview
  • box/card/panel

This was the moment Erire stopped being just a parser experiment and became an actual desktop UI runtime.

Milestone 3: Source-Order Runtime + Event Model

The current runtime executes statements in strict source order. That decision shaped the whole framework:

  • UI nodes exist immediately after screen.add[...]
  • variables mutate immediately
  • events observe the state produced by earlier statements
  • onLoad[...] is deferred so initial hydration can happen after app creation

This keeps the mental model simple and debuggable.

Milestone 4: Native Logic Layer in .er

The addition of:

  • if
  • else.if
  • else
  • while
  • for
  • native text.*, math.*, sys.*, time.now[]

gave Erire a real "backend inside the UI language" story for lightweight logic.

That change matters because not every app needs Python for basic control flow and derived values.

Milestone 5: Automatic backend.er

The frontend loader now automatically merges backend.er before the entry module when both live in the same project area.

This created a practical split:

  • backend.er: setup, derived values, native control flow
  • main.er: screen creation and UI composition

That simple convention is one of the most useful design decisions in the current codebase.

Milestone 6: Python Bridge and Structured Results

The Python bridge turned Erire into a much more capable app platform.

Key effects:

  • import python "...py" as alias
  • py.call["alias.function"; ...]
  • JSON-encoded argument passing
  • structured JSON responses mapped into dotted runtime variables

This is the milestone that made file tools, dashboards, automation apps, and bridge-driven workflows practical.

Milestone 7: Packaging and App Distribution

The packager moved Erire from "run scripts locally" to "ship desktop apps".

Current packaging behavior includes:

  • bundling the entry app and backend.er
  • bundling assets/, python/, and app.json
  • compiling Python helpers with PyInstaller when needed
  • optionally compiling native helpers from cpp/
  • applying icons to the output .exe
  • appending the payload to ErireRunner.exe

This is the milestone that made Erire an application workflow rather than just a runtime.

Milestone 8: UI Depth Features

The runtime continued to add the kinds of features real desktop apps need:

  • WebView controls and browser commands
  • image nodes and window branding
  • multiline and read-only inputs
  • custom-drawn presets
  • pages via page[...] + screen.show[...]
  • animations
  • console profile launching

This phase is where the examples began to look like tools and dashboards instead of toy demos.

Milestone 9: Erire Studio

Erire Studio is a major shift in the project:

  • dedicated project workflow
  • structured templates
  • file tree, editor, outline, logs
  • run/check/build integration
  • Python workflow awareness

Studio changes the developer story from "edit .er files manually" to "build complete projects inside a purpose-built environment".

Milestone 10: Design Work for vNext

The roadmap docs show a clear next direction:

  • layout primitives
  • reusable components
  • reactive state
  • stronger styling/theming
  • richer project structure

These ideas live today in docs/erire-next-language.md. They are important because they explain where the current system wants to go without pretending those features already exist.

Multiline And Read-Only Inputs

Erire Bytecode Studio needed two UI capabilities that were missing from the runtime:

  • pasting and editing multiple lines of text
  • showing long generated output without letting the user modify it

To support that cleanly inside Erire itself, the runtime now supports these input properties:

  • input.multiline["true"]
  • input.readonly["true"]
  • input.readOnly["true"]

Why This Was Added

Erire already had input, but it was effectively a single-line field. That made it awkward to build:

  • text editors
  • byte/hex viewers
  • log panes
  • reverse-decoding tools
  • settings areas with larger text blocks

Adding multiline and read-only behavior strengthens Erire as a desktop UI language instead of pushing that work into external tooling.

Syntax

screen.add[
    input.value["Line 1\nLine 2"]
        .multiline["true"]
        .x[40]
        .y[80]
        .w[420]
        .h[180]
]

screen.add[
    input.value["Generated output"]
        .multiline["true"]
        .readonly["true"]
        .x[40]
        .y[280]
        .w[420]
        .h[180]
]

Runtime Behavior

  • multiline inputs use a real multiline Windows EDIT control
  • Enter inserts a new line
  • vertical scrolling is enabled automatically
  • read-only inputs remain selectable and copyable while blocking edits
  • larger default heights are applied when multiline is enabled

Where It Was Implemented

  • runtime property parsing: src/runtime.c
  • semantic validation: src/semantic.c
  • node spec / state: src/ui.h
  • Win32 input creation and layout: src/ui.c

What Each Line Does

screen.create[...]

Creates the window. In current examples, the six-argument form is the common style:

screen.create[app; size; 100; 100; 820; 520]

Only the last four values matter to placement and size here:

  • 100, 100: initial window position
  • 820, 520: initial window size

screen.title[...]

Sets the window caption.

screen.bg[...]

Sets the window background color.

screen.add[...]

Adds one UI element.

means:

  • create a text node
  • set its text to "Hello from Erire"
  • give it id headline
  • place it at (30, 30)
  • size it to (360, 40)
  • draw it with font size 26
  • draw it in white

onClick[...]

The button event block runs when the user clicks the button.

screen.setText["headline"; "Button clicked"]

That line finds the node whose id is headline and updates its content.

Most Erire apps start with this pattern:

  1. Create the window.
  2. Add nodes.
  3. Give important nodes ids.
  4. Update those ids inside events.

Key Concepts

bind[...]

This line:

input.bind["note_text"]

means the input pushes its current text into the variable $note_text.

In the current runtime:

  • the initial input value is copied into $note_text
  • every later change updates $note_text
  • if onChange[...] exists, the bound variable is updated before the event body runs

onChange[...]

onChange[
    screen.setText["livePreview"; $note_text]
]

This gives you instant UI feedback without needing Python or extra state machinery.

multiline["true"]

Turns the input into a multiline text area.

readonly["true"]

Keeps the field selectable and copyable, but blocks editing.

This is ideal for:

  • generated output
  • logs
  • previews
  • read-only content panes

Why This Matters

Bindings and onChange[...] are the fastest way to make Erire feel alive.

Use them for:

  • search boxes
  • filters
  • settings forms
  • live previews
  • editor-like tools

A Good Practical Rule

Use the input itself for authoring, and a read-only multiline input for output. That pattern maps well to:

  • code generators
  • note tools
  • JSON formatters
  • prompt tools
  • admin utilities

.page["home"]
.page["docs"]

Only the active page is shown.

screen.show[...]

This is the simplest clean way to build multi-screen apps in current Erire.

webview.url[...]

webview.url["https://www.example.com"].id["browser"]

  • the app would become crowded as one screen

Use a webview when:

  • you need live web content
  • you want an internal browser shell
  • you want to embed a docs/help surface
  • you want hybrid desktop + web content

Step 2: Create python/app_bridge.py

This script follows the bridge contract used by the current runtime.

#!/usr/bin/env python3
import argparse
import base64
import json
from pathlib import Path
from typing import Any


PROJECT_ROOT = Path(__file__).resolve().parent.parent


def _emit(payload: Any, exit_code: int = 0) -> None:
    print(json.dumps(payload, ensure_ascii=False))
    raise SystemExit(exit_code)


def _decode_args(encoded: str) -> list[Any]:
    if not encoded:
        return []
    decoded = base64.b64decode(encoded.encode("ascii")).decode("utf-8")
    value = json.loads(decoded)
    if not isinstance(value, list):
        raise ValueError("Bridge arguments must decode to a JSON list")
    return value


def ping() -> dict[str, Any]:
    return {
        "message": "Python bridge connected",
        "root": str(PROJECT_ROOT),
    }


def make_summary(name: Any = "") -> dict[str, Any]:
    text = str(name).strip() or "Unnamed Project"
    return {
        "message": "Summary generated",
        "title": f"Project: {text}",
        "body": f"{text} is now flowing through the Erire <-> Python bridge.",
    }


ALLOWED_FUNCTIONS = {
    "ping": ping,
    "make_summary": make_summary,
}


def __erire_dispatch__(function_name: str, arguments: list[Any]) -> Any:
    handler = ALLOWED_FUNCTIONS.get(function_name)
    if handler is None:
        raise PermissionError(f"Function '{function_name}' is not whitelisted")
    return handler(*arguments)


def main() -> None:
    parser = argparse.ArgumentParser(add_help=False)
    parser.add_argument("--erire-call", required=True)
    parser.add_argument("--erire-args-b64", default="")
    args = parser.parse_args()

    try:
        bridge_args = _decode_args(args.erire_args_b64)
        result = __erire_dispatch__(args.erire_call, bridge_args)
        _emit(result, 0)
    except Exception as exc:
        _emit({"status": "error", "error": str(exc)}, 1)


if __name__ == "__main__":
    main()

Why The Script Looks Like This

The current runtime does not import Python as a library. It launches the bridge script as a subprocess and passes:

  • --erire-call <function_name>
  • --erire-args-b64 <base64(json_list)>

So the script must:

  1. parse those flags
  2. decode the argument list
  3. dispatch to a whitelisted function
  4. print a JSON result

That is the contract.

Tutorial 5: Build and Package

This tutorial explains how to go from a working .er app to a packaged Windows executable.

Goal

We will cover:

  • app.json

  • building an .exe
  • icons

Step 1: Add app.json

Create an app.json beside your project entry area.

Example:

{
  "name": "Bridge Demo",
  "entry": "src/main.er",
  "output": "build/bridge_demo.exe",
  "icon": "assets/brand/app.ico",
  "win_title": "Bridge Demo",
  "win_icon": "assets/brand/app.ico",
  "version": "0.1.0"
}

Useful fields today:

  • name
  • entry
  • output
  • icon
  • win_title
  • win_icon

Important current behavior:

  • runtime and packaging metadata are read beside the entry .er file
  • Studio project discovery uses a root project manifest

If your real entry is src/main.er, keep this nuance in mind when deciding where the runtime-facing app.json should live.

Step 2: Validate Before Packaging

Always run:

.\erire.exe --check src\main.er

This catches syntax and semantic problems before the build pipeline does more work.

Step 3: Build The App

.\erire.exe --build src\main.er build\bridge_demo.exe

If you do not pass an output path, Erire writes to:

dist\<entry-name>.exe

Step 4: Override Icon Or Window Title At Build Time

You can override packaging metadata from the CLI:

.\erire.exe --build --icon=assets\brand\app.png --win_title="Bridge Demo" src\main.er build\bridge_demo.exe

Current icon notes:

  • .ico and .png are supported
  • PNG icons are converted through tools/png_to_ico.py

What The Packager Bundles

The current packager includes:

  • the entry .er file
  • auto-loaded backend.er
  • imported Python bridges
  • assets/
  • python/
  • app.json
  • optional compiled helper from cpp/

That means good project structure pays off immediately.

Python Packaging Requirement

If your app uses Python imports, packaging expects PyInstaller:

python -m pip install pyinstaller

Without it, packaging a Python-backed app will fail.

C++ Helper Support

If your project contains cpp/ sources, the packager can compile them into:

cpp\native_cpp.exe

This is optional, but it gives you a path for native helper workflows.

What Happens At Runtime

The final executable is based on ErireRunner.exe.

During packaging:

  • the runner is copied
  • the packaged app payload is appended to it

During launch:

  • the bundled payload is extracted to a temporary directory
  • the entry app is then run from that extracted package area

A Safe Build Checklist

Before shipping:

  1. Run --check.
  2. Launch the source app once.
  3. Confirm every imported Python file exists.
  4. Confirm assets/ contains everything the UI expects.
  5. Confirm app.json title and icon metadata.
  6. Confirm PyInstaller is installed if Python is used.
  7. Build the .exe.
  8. Launch the packaged .exe and test the real packaged flow.

Common Build Commands

Build with default output

.\erire.exe --build src\main.er

Build with explicit output

.\erire.exe --build src\main.er build\my_app.exe

Build with icon override

.\erire.exe --build --icon=assets\brand\app.ico src\main.er build\my_app.exe

Build with icon and window title override

.\erire.exe --build --icon=assets\brand\app.png --win_title="My App" src\main.er build\my_app.exe

Erire Next Language Design

Executive Summary

Erire vNext should stay a UI-only language, but move from an absolute-positioned "screen script" into a declarative, component-oriented UI language with:

  • reusable components
  • flexible layouts instead of raw x/y everywhere
  • reactive state
  • reusable styles and themes
  • stronger event composition
  • project-level imports and modular structure
  • fast incremental rendering on top of the current C runtime

The identity to preserve is:

  • chain-based syntax
  • readable bracket calls
  • UI-first mental model
  • simple integration with Python via py.call

The design below keeps existing syntax valid while introducing a cleaner "vNext layer" above it.

1. Component System

Concept

Components give Erire reusable UI building blocks without introducing classes or OOP. A component is a named UI template with props and optional local state.

Syntax

component Card[title; value] [
    box.preset["card"]
       .children[
           text.value[$title].preset["cardTitle"]
           text.value[$value].preset["metric"]
       ]
]

Component usage inside screen.add or another component:

screen.add[Card["CPU"; "37%"]]

Named props for readability:

screen.add[Card.title["CPU"].value["37%"]]

Example 1

component MetricTile[label; number] [
    card.style["metricCard"]
        .children[
            text.value[$label].style["mutedLabel"]
            text.value[$number].style["heroMetric"]
        ]
]

screen.add[MetricTile["Memory"; "8.2 GB"]]
screen.add[MetricTile["Disk"; "72%"]]

Example 2

component ToolbarButton[text; action] [
    button.text[$text]
          .style["toolbarBtn"]
          .onClick[
              event.emit[$action]
          ]
]

screen.add[ToolbarButton["Build"; "build.requested"]]

Integration With Existing Erire

  • screen.add[...] still stays the root composition point.
  • Existing elements remain legal.
  • A component expands into the same internal element tree the runtime already understands.
  • Existing preset and event concepts become building blocks inside components.

2. Layout System

Concept

Layout is the most important evolution. Erire should stop forcing every serious app to hand-place everything. The vNext layout model should provide row, column, stack, and grid, while still allowing absolute positioning when needed.

Syntax

layout.column[
    gap[12]
    padding[20]
    children[
        text.value["Hello"]
        button.text["Click"]
    ]
]

Core Layout Nodes

  • layout.row[...]
  • layout.column[...]
  • layout.stack[...]
  • layout.grid[...]
  • layout.scroll[...]

Shared Layout Properties

  • gap[...]
  • padding[...]
  • align[start|center|end|stretch]
  • justify[start|center|between|around|end]
  • grow[...]
  • basis[...]
  • minW[...], minH[...]
  • maxW[...], maxH[...]
  • fillX[true|false], fillY[true|false]

Example 1

screen.add[
    layout.column[
        padding[24]
        gap[16]
        children[
            text.value["System Dashboard"].style["pageTitle"]
            layout.row[
                gap[16]
                children[
                    MetricTile["CPU"; $snapshot.cpu.usage_label]
                    MetricTile["RAM"; $snapshot.ram.used_label]
                    MetricTile["Disk"; $snapshot.disk.used_label]
                ]
            ]
        ]
    ]
]

Example 2

screen.add[
    layout.grid[
        columns[2]
        gap[18]
        children[
            card.children[
                text.value["Project"]
                input.bind["projectName"]
            ]
            card.children[
                text.value["Python Path"]
                input.bind["pythonPath"]
            ]
        ]
    ]
]

Integration With Existing Erire

  • Existing .x/.y/.w/.h remain valid for low-level placement or custom overlays.
  • Layout nodes are just new element kinds in the parser/runtime.
  • Child layout resolution runs before native control positioning.
  • The runtime can still lower final layout to Win32 rectangles, so the rendering backend remains incremental, not a rewrite.

3. State Management

Concept

Reactive state lets the UI update automatically when values change, instead of manually calling screen.setText for every change.

Syntax

state["count"; 0]

State mutation:

state.set["count"; $count + 1]

Derived state:

state.computed["summary"; $firstName + " " + $lastName]

Example 1

state["count"; 0]

screen.add[
    layout.column[
        gap[10]
        children[
            text.value["Counter: " + $count]
            button.text["Increment"].onClick[
                state.set["count"; $count + 1]
            ]
        ]
    ]
]

Example 2

state["search"; ""]

screen.add[
    input.bind["search"]
         .placeholder["Search docs"]
]

screen.add[
    text.value["Searching for: " + $search]
]

Integration With Existing Erire

  • var.set still exists for one-shot values and bridge results.
  • state[...] is for reactive UI values.
  • input.bind[...] becomes state-aware automatically.
  • screen.setText remains available for imperative escape hatches.

4. Styling System

Concept

Presets were a good first step. vNext should formalize them into reusable styles and themes with inheritance.

Syntax

style["primaryBtn";
    bg:"#2563eb";
    color:"#ffffff";
    radius:12;
    padding:14
]

Theme definition:

theme["light";
    bg:"#f8fafc";
    surface:"#ffffff";
    text:"#111827";
    muted:"#64748b";
    accent:"#2563eb"
]

Example 1

style["sidebarItem";
    padding:12;
    radius:10;
    color:"#334155";
    bg:"#ffffff"
]

screen.add[
    button.text["Files"].style["sidebarItem"]
]

Example 2

theme["dark";
    bg:"#0f172a";
    surface:"#111827";
    text:"#f8fafc";
    muted:"#94a3b8";
    accent:"#38bdf8"
]

Integration With Existing Erire

  • preset can stay as shorthand or alias for style.
  • Existing .bg/.color/.radius/.padding remain override properties.
  • Theme values can flow into the current custom-drawn runtime theme model.

5. Advanced Events

Concept

Erire already supports event blocks. vNext should make them composable and reactive without becoming a full scripting language.

Syntax

button.text["Save"].onClick[
    if[$isValid][
        state.set["saving"; true]
        py.call["api.save"; $form]
        event.emit["toast.success"; "Saved"]
    ]
]

New event helpers:

  • event.emit[...]
  • event.listen[...]
  • event.preventDefault[]
  • event.stop[]

Example 1

button.text["Run"].onClick[
    screen.show["output"]
    event.emit["build.run"]
]

event.listen["build.run"][
    state.set["status"; "Running..."]
]

Example 2

input.bind["query"].onChange[
    if[$query != ""][
        state.set["searching"; true]
        var.set["result"; py.call["docs.search"; $query]]
        state.set["searching"; false]
    ]
]

Integration With Existing Erire

  • onClick, onChange, onLoad stay.
  • Event blocks continue to accept multiple statements.
  • Existing if[...] model stays the conditional primitive.

8. Performance Model

Parsing Model

  • Lexer remains token-based and fast.
  • Parser adds new nodes for component, layout.*, style, theme, and state.
  • Component expansion can happen in semantic lowering.
  • Imports should be resolved into one module graph before runtime execution.

Rendering Pipeline

  1. Parse source into AST
  2. Resolve imports/components/styles/themes
  3. Build a lightweight virtual UI tree
  4. Compute layout tree
  5. Diff against the currently mounted tree
  6. Reposition or update only changed Win32/native nodes

Efficient Updates

  • State changes mark dependent nodes dirty
  • Layout is recalculated only for affected branches
  • Text/style changes update only matching node handles
  • Webview/image/input keep stable native handles where possible

Integration With Existing Erire

  • Reuses the current node registry, page system, ids, and event bindings
  • Adds a diff/layout phase before the existing native drawing layer
  • Avoids a full runtime rewrite

Step-By-Step Evolution Path

  1. Add style and theme syntax first because they extend the existing preset system.
  2. Add state[...] and state.set[...] with dependency tracking.
  3. Add layout.row/column/stack/grid and child containers.
  4. Add component definitions and expansion.
  5. Add import erire module support.
  6. Add event.emit and event.listen.
  7. Add erire dev live reload and diagnostics.

Why Erire Is Easier Than Traditional UI Frameworks

  • It starts from UI, not from application plumbing.
  • A small number of chain-based commands covers most desktop app work.
  • Python bridge support removes the need to invent a second backend story.
  • The mental model is simpler than Flutter widgets or React component lifecycle.
  • The runtime can stay native and fast because the language is specialized for desktop UI only.
